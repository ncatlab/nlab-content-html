Idea

In computer science, a monad describes a ‚Äúnotion of computation‚Äù. Formally, it is a map that

  • sends every type XX of some given programming language to a new type T(X)T(X) (called the ‚Äútype of TT-computations with values in XX‚Äù);

  • is equipped with a rule for composing two functions of the form f:X‚ÜíT(Y)f : X \to T(Y) (called Kleisli functions) and g:Y‚ÜíT(Z)g : Y \to T(Z) to a function g‚àòf:X‚ÜíT(Z)g \circ f : X \to T (Z) (their Kleisli composition);

  • is in a way that is associative in the evident sense and unital with respect to a given unit function called pure X:X‚ÜíT(X)pure_X : X \to T(X), to be thought of as taking a value to the pure computation that simply returns that value.

This is essentially the same structure as a monad in category theory, but presented differently; see below for the precise relationship.

For imperative programs in functional programming

Monads provide one way to ‚Äúembed imperative programming in functional programming‚Äù, and are used that way notably in the programming language Haskell. But monads, as well as comonads and related structures, exist much more generally in programming languages; an exposition is in (Harper). For an account of the use of monads in industry see Benton15, pp. 11-12.

For instance when the monad T(‚àí)T(-) forms product types T(X)‚âîX√óQT(X) \coloneqq X \times Q with some fixed type QQ that carries the structure of a monoid, then a Kleisli function f:X‚ÜíY√óQf : X \to Y \times Q may be thought of as a function X‚ÜíYX \to Y that produces a side effect output of type QQ. The Kleisli composition of two functions f:X‚ÜíY√óQf \colon X \to Y \times Q and g:Y‚ÜíZ√óQg \colon Y \to Z \times Q then not only evaluates the two programs in sequence but also combines their QQ-output using the monoid operation of QQ; so if fx=(y,q)f x = (y,q) and gy=(z,q‚Ä≤)g y = (z,q') then the final result of (g‚àòf)(x)(g \circ f)(x) will be (z,qq‚Ä≤)(z, q q'). For example, QQ might be the set of strings of characters, and the monoid operation that of concatenation of strings (i.e. QQ is the free monoid on the type of characters). If the software is designed such that values of type QQ computed in this way appear on the user‚Äôs screen or are written to memory, then this is a way to encode input/output in functional programming (see the IO monad? below).

But monads have plenty of further uses. They are as ubiquituous (sometimes in disguise) in computer science as monads in the sense of category theory are (sometimes in disguise) in category theory. This is no coincidence, see Relation to monads in category theory below.

Relation to monads in category theory

In computer science, a programming language may be formalised or studied by means of a category, called the syntactic category ùíû\mathcal{C}, whose

  • objectsX‚ààùíûX \in \mathcal{C} are the types of the language,

  • morphismsX‚ÜíYX \to Y are the terms or programs (or an equivalence class of such) that takes a value of type XX as input and returns a value of type YY.

This point of view (see computational trinitarianism) is particularly useful when studying purely functional programming languages.

Under this relation between type theory and category theory monads on the type system in the sense of computer science are monads in the sense of category theory, being certain endofunctors

T:ùíû‚ü∂ùíû T \colon \mathcal{C} \longrightarrow \mathcal{C}

on the syntactic category. This functor

  1. sends each type, hence object X‚ààùíûX \in \mathcal{C} to another object T(X)T(X);

  2. the unit natural transformation œµ:Id ùíû‚áíT\epsilon \colon Id_{\mathcal{C}} \Rightarrow T of the monad TT provides for each type XX a component morphism pure X:X‚ÜíT(X)pure_X : X \to T(X);

  3. the multiplication natural transformation Œº:T‚àòT‚áíT\mu \colon T \circ T \Rightarrow T of the monad provides for each object XX a morphism Œº X:T(T(X))‚ÜíT(X)\mu_X : T(T(X)) \to T(X) which induces the Kleisli composition by the formula

Here the morphism T(g)T(g) in the middle of the last line makes use of the fact that T(‚àí)T(-) is indeed a functor and hence may also be applied to morphisms / functions between types. The last morphism Œº(Z)\mu(Z) is the one that implements the ‚ÄúTT-computation‚Äù.

The monads arising this way in computer science are usually required also to interact nicely with the structure of the programming language, as encoded in the structure of its syntactic category; in most cases, terms of the language will be allowed to take more than one input, so the category ùíû\mathcal{C} will be at least monoidal, and the corresponding kind of ‚Äònice‚Äô interaction corresponds to the monad‚Äôs being a strong monad.

The ‚Äòbind‚Äô operation is a means of describing multiplication on such a strong monad MM. It is a term of the form MA‚Üí(MB) A‚ÜíMBM A \to (M B)^A \to M B, which is equivalent to a map of the form MA√óMB A‚ÜíMBM A \times M B^A \to M B. It is the composite

MA√óMB A‚ÜístrengthM(A√óMB A)‚ÜíMeval A,MBMMB‚ÜímBMB M A \times M B^A \stackrel{strength}{\to} M(A \times M B^A) \stackrel{M eval_{A, M B}}{\to} M M B \stackrel{m B}{\to} M B

where m:MM‚ÜíMm \colon M M \to M is the monad multiplication.

When monads are defined in Haskell, the Kleisli composition, ‚Äòbind‚Äô, is defined in Haskell. So monads in Haskell are always enriched monads, according to the self-enrichment defined by the function type in Haskell.

Examples

Various monads are definable in terms of the the standard type-forming operations (product type, function type, etc.). These include the following.

  • A functional program with input of type XX, output of type YY and mutable state SS is a function (morphism) of type X√óS‚ü∂Y√óSX \times S \longrightarrow Y \times S. Under the (Cartesian product ‚ä£\dashv internal hom)-adjunction this is equivalently given by its adjunct, which is a function of type X‚ü∂[S,S√óY]X \longrightarrow [S, S \times Y ]. Here the operation [S,S√ó(‚àí)][S, S\times (-)] is the monad induced by the above adjunction and this latter function is naturally regarded as a morphism in the Kleisli category of this monad. This monad [S,S√ó(‚àí)][S, S\times (-)] is called the state monad for mutable states of type S.

  • The maybe monad is the operation X‚Ü¶X‚àê*X \mapsto X \coprod \ast. The idea here is that a function X‚ü∂YX \longrightarrow Y in its Kleisli category is in the original category a function of the form X‚ü∂Y‚àê*X \longrightarrow Y \coprod \ast so either returns indeed a value in YY or else returns the unique element of the unit type/terminal object *\ast. This is then naturally interpreted as ‚Äúno value returned‚Äù, hence as indicating a ‚Äúfailure in computation‚Äù.

  • The continuation monad for a given type SS acts by X‚Ü¶[[X,S],S]X \mapsto [[X,S],S].

  • A number of further monads are similarly definable in terms of standard type-forming operations, such as the reader monad and the writer comonad.

    Given a type WW, then the reader monad is the operation of forming the function type [W,‚àí]=(W‚Üí(‚àí))[W,-] = (W\to (-)); the writer comonad is the operation of forming the product type W√ó(‚àí)W\times (-) and the composite of writer followed by reader is the state monad [W,W√ó(‚àí)][W, W \times (-)].

    When WW carries the structure of a monoid object then writer also inherits the structure of a monad (on top of being a comonad) and converse for reader.

Other monads may be supplied ‚Äúaxiomatically‚Äù by the programming language, This includes

  • the IO monad? in Haskell.

  • The completion monad may be used, as in constructive analysis, for dealing for instance with real numbers.

  • Equipping homotopy type theory (say implemented as a programming language concretely in Coq or Agda) with two axiomatic idempotent monads, denoted ‚ôØ\sharp and Œ†\Pi, with some additional data and relations, turns it into cohesive homotopy type theory. See also modal type theory.
