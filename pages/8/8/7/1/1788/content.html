In programming it frequently happens that a program (say prog 12prog_{12}) with nominal output data type D 2D_2 de facto outputs data of some modified type T(D)T(D) due to “external effects”, where

(1)D↦T(D) D \;\mapsto\; T(D)

is some general operation sending data types DD to new data types T(D)T(D).

According to the intended behaviour of these programs, it remains to specify how exactly prog 23[−]prog_{23}[-] “carries T(−)T(-)-effects along”, hence what the “bind” operation really is.

For instance, in the above example where T(D 2)=D 2×StringT(D_2) = D_2 \times String, the evident way is to use the concatenation String×String→concatStringString \times String \xrightarrow{\; concat \;} String and set:

But whatever design choice one makes for how to “carry along effects”, it must be consistent in that applying the method to a triple of T(−)T(-)-effectful programs which are nominally composable, then their effectful composition should be unambiguously defined in that it is associative, satisfying the following “law” (equation):

(3)prog 34[prog 23[prog 12(−)]]=(prog 34[prog 23(−)])[prog 12(−)]. prog_{34}\Big[ prog_{23}\big[ prog_{12}(-) \big] \Big] \;\;\;=\;\;\; \Big( prog_{34}\big[ prog_{23}(-) \big] \Big) \big[ prog_{12}(-) \big] \,.

Finally, for such a notion of effectful programs to be usefully connected to “pure” programs without effects, it ought to be the case that for any program prog:D 0→D 1prog \,\colon\, D_0 \xrightarrow{\;} D_1 that happens to have no T(−)T(-)-effects, we have a prescription for how to regard it as a T(−)T(-)-effectful program in a trivial way. For that purpose there should be defined an operation

(4)ret D 1:D 1→T(D 1) ret_{D_1} \;\colon\; D_1 \xrightarrow{\;} T(D_1)

which does nothing but “return” data of type D 1D_1, but re-regarded as effectful T(D 1)T(D_1)-data in a trivial way; so that we may construct the trivially effectful program ret D 1(prog 01(−)):D 0→T(D 1)ret_{D_1}\big(prog_{01}(-)\big) \;\colon\; D_0 \xrightarrow{\;} T(D_1).

For instance, in the above example of log-message effects this would be the operation D→D×StringD \to D \times String which assigns the emtpty string d↦(d,∅)d \mapsto (d, \varnothing).

The final consistency condition then is that “carrying along trivial effects is indeed trivial”, i.e. that

(5)prog 01[ret D 0(−)]=prog 01andret D 1[prog 01(−)]=prog 01. prog_{01} \big[ ret_{D_0}(-) \big] \;=\; prog_{01} \;\;\;\;\;\;\;\;\; \text{and} \;\;\;\;\;\;\;\;\; ret_{D_1}\big[ prog_{01}(-) \big] \;=\; prog_{01} \,.

Notice that the associativity condition (3) and the unitality condition (5) jointly say that data types with T(−)T(-)-effectful programs between them, in the above sense, form a category. In category theory this is known as the Kleisli category of a monad TT.

Traditionally in category theory, the axioms on monads are presented in a slightly different way, invoking a monad “product” natural transformation T∘T→μTT \circ T \xrightarrow{ \mu } T instead of the “binding” operation. One readily checks that these two axiomatic presentations of monads are in fact equal.

In summary, a choice of assignments

  1. D↦T(D)D \;\mapsto\; T(D) \;—\; of types of effectful data of nominal type DD (1),

  2. D↦ret DD \;\mapsto\; ret_D \;—\; regarding plain DD-data as trivially effectful (4),

  3. prog↦prog[−]prog \;\mapsto\; prog[-] \;—\; specifying how to execute a given progprog while carrying along any previous T(−)T(-)-effects (2)

subject to

  1. the associativity condition (3)

  2. the unitality condition (5)

is called a monad in computer science and serves to encode the notion that all programs may be subject to certain external effects.

The running example above is known as the writer monad, since it encodes the situation where programs may have the additional effect of writing a message string into a given buffer.
