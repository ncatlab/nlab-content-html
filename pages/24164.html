Idea

While homotopy type theory formalizes homotopy theory, it is not a priori clear ‚Äì and in fact is or was an open problem ‚Äì how to formalize general homotopy-coherent structures of higher algebra/higher category theory: Since these typically involve an infinite hierarchy of coherence-conditions, these cannot be axiomatized directly, but one needs some scheme that generates them. This turned out to be subtle.

Eric Finster had previously considered another variant of type theory, called opetopic type theory which natively talks about infinity-categories and their higher coherences by type-theoretically formalizing the structure of opetopic sets. In new work Finster 18 he gives something like an implementation of aspects of opetopic type theory within homotopy type theory and provides evidence that this yields a tool to solve the general problem of coherences of higher algebra/higher category theory within homotopy type theory.

Polynomials serve as a notion of ‚Äúhigher signature‚Äù. Following ideas from the categorical approach to universal algebra, we are going to encode the relations or axioms of our structure using a monadic multiplication on P.

Definitions and Examples

Here we collect the definitions and ideas that were given in Eric‚Äôs talk. We can also use the Agda formalisation for reference too.

Definition

Fix a type II of sorts. A polynomial over II, PolyI\Poly I, is the data of

  • A family of operations Op:I‚Üíùí∞\Op : I \to \mathcal{U}
  • For each operation, for each i:Ii : I, a family of sorted parameters
    Param i:(f:Opi)‚ÜíI‚Üíùí∞\Param_i : (f : \Op i) \to I \to \mathcal{U}
Remark

  • For i:Ii : I, an element f:Opif: \Op i represents an operation whose output sort is ii.

  • For f:Opif : \Op i and j:Ij : I, and element p:Param i(f,j)p : \Param_i(f, j) represents an input parameter of sort jj.

  • The Opi\Op i and Param i(f,j)\Param_i (f, j) are not truncated at set level. So operations and parameters can have higher homotopy.

  • In HoTT book notation, we can write the previous definition as:

PolyI‚â°‚àë Op:I‚Üíùí∞‚àè i:IParam i\Poly I \equiv \sum_{\Op : I \to \mathcal{U}}\prod_{i : I} \Param_i
Param i‚â°‚àè f:OpiI‚Üíùí∞\Param_i \equiv \prod_{f : \Op i}I \to \mathcal{U}
Example

We start with the simplest non-trivial example, when I‚â°1I \equiv \mathbf{1} the unit type. Looking at Poly1\Poly \mathbf{1} we clearly have Op:1‚Üíùí∞‚âÉùí∞\Op : \mathbf{1} \to \mathcal{U}\simeq \mathcal{U}. So Op\Op is simply a type. Next we have Param:(f:Op)‚Üí1‚Üíùí∞‚âÉOp‚Üíùí∞\Param : (f : \Op) \to \mathbf{1} \to \mathcal{U}\simeq \Op \to \mathcal{U} so Param\Param is just a family of types indexed over the type Op\Op.

Now if Op‚â°Fin 3\Op \equiv \Fin_3 then there would be 3 operations. Take one of these operations and call it o 1o_1, Param\Param would then have to chose what the type of parameters should be, let‚Äôs choose Fin 2\Fin_2. This would mean that o 1o_1 would be an operation (o 1:Opo_1 : \Op) with two parameters p:Fin 2‚â°Paramo 1p : \Fin_2 \equiv \Param o_1. Note o 1o_1 is very much like the operator in an abstract syntax tree in this case.

When the type of sorts II gets more structure, the objects we discuss get complicated quickly. Even letting Paramo 1\Param o_1 be some arbitrary type gives us an interesting object.
